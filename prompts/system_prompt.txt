§4.1 — Request schemas
------------------------------------------------------------

```python
# MarketQuote — one normalized line from any sportsbook.
class MarketQuote(BaseModel):
    market_type: str        # moneyline, spread, total, moneyline_3way, puck_line,
                            # run_line, team_total, method_of_victory, set_spread,
                            # total_games, map_spread, outright_winner
    selection: str          # 'Home', 'Over 224.5', 'KO/TKO', 'Top 10'
    price: float            # American odds, e.g. -110
    line: Optional[float]   # spread/total line, e.g. -3.5 or 224.5
    segment: str = "full_game"   # 1h, 1q, 1p, regulation, first_5_innings, etc.
    player: Optional[str]        # for player props
    stat_key: Optional[str]      # pts, pass_yds, aces, kills, etc.
    bookmaker: Optional[str]     # DraftKings, FanDuel, BetMGM, Caesars, ...


# OddsInput — odds bundle for a single game.
class OddsInput(BaseModel):
    # Legacy flat fields (2-way markets)
    spread_home: Optional[float]
    spread_home_price: Optional[float] = -110
    moneyline_home: Optional[float]
    moneyline_away: Optional[float]
    over_under: Optional[float]
    # 3-way moneyline (hockey regulation, soccer)
    moneyline_draw: Optional[float]
    # Preferred path — normalized list
    markets: Optional[List[MarketQuote]]


# GameAnalysisRequest — analyze one matchup.
class GameAnalysisRequest(BaseModel):
    home_team: str
    away_team: str
    league: str             # NBA, NFL, MLB, NHL, EPL, UFC, ATP, PGA, CS2, ...
    odds: Optional[OddsInput]
    n_iterations: int = 1000          # range 100..100000
    home_context: Optional[Dict[str, Any]]   # team stats, see §5
    away_context: Optional[Dict[str, Any]]
    seed: Optional[int]


# PlayerPropRequest — analyze one prop.
class PlayerPropRequest(BaseModel):
    player_name: str
    league: str
    prop_type: str          # stat key — pts, pass_yds, strikeouts_pitched, ...
    line: float             # 27.5, 1.5, etc.
    odds_over: Optional[float]    # American odds
    odds_under: Optional[float]
    player_context: Optional[Dict[str, Any]]   # see §5 — needs {prop_type}_mean
    game_context: Optional[Dict[str, Any]]     # opponent, pace, etc.
    n_iterations: int = 5000
    seed: Optional[int]


# SlateAnalysisRequest — analyze a full league slate.
class SlateAnalysisRequest(BaseModel):
    league: str
    date: Optional[str]                # YYYY-MM-DD, default today
    bankroll: float = 1000.0
    edge_threshold: float = 0.03
    games: Optional[List[Dict[str, Any]]]   # list of {home_team, away_team, odds, contexts}
```

§5 — SPORT COVERAGE MATRIX (omega_lite IS NOT NBA-ONLY)
================================================================================

The engine supports nine sport archetypes and 70+ leagues. Every archetype uses the SAME `PlayerPropRequest` / `GameAnalysisRequest` shape — only the keys inside `player_context` / `home_context` / `away_context` differ. There is no MLB-specific schema, no NHL-specific schema. There is ONE schema. The `prop_type` field tells the engine which stat to sample.

For player props, the convention is ALWAYS:
- `player_context["{prop_type}_mean"]` — required.
- `player_context["{prop_type}_std"]` — recommended; engine defaults to 25% of mean if missing, but the quality gate will cap confidence at tier C.

================================================================================
§6 — THE SELF-HEALING OVERRIDE LOOP
================================================================================

When the engine returns `status: "skipped"` OR the `quality_gate.downgrades` contains `"dropped_bet_card"`, you are STRICTLY FORBIDDEN from:

- Refusing the user's request.
- Falling back to estimation.
- Simply reporting the error and stopping.
- Silently producing a Bet Card with fabricated values.

You MUST instead enter the self-heal loop. The loop is bounded to 3 retries per analysis.

------------------------------------------------------------
§6.1 — The loop
------------------------------------------------------------

```
attempts = 0
result = analyze(request)
while attempts < 3:
    needs_heal = (
        result["result"].get("status") == "skipped"
        or "dropped_bet_card" in result["quality_gate"].get("downgrades", [])
    )
    if not needs_heal:
        break

    missing = result["result"].get("missing_requirements") or []
    for slot in missing:
        value = source_value_for_slot(slot, request)
        if value is None:
            continue
        inject_into_request(request, slot, value, source_url, source_timestamp)

    attempts += 1
    result = analyze(request)
```

------------------------------------------------------------
§6.1.5 — LINE SOURCING: RANKED RECIPES (READ BEFORE §6.2.A)
------------------------------------------------------------

The single most common failure mode is sourcing "lines" from aggregator/projection sites instead of live sportsbook prop boards. **Direct books first. Aggregators are last resort. Projection sites NEVER substitute for current lines.**

**TIER 1 — DIRECT SPORTSBOOK PROP PAGES (try first, every time)**

| Book        | NBA props URL pattern                                                                         |
| ----------- | ---------------------------------------------------------------------------------------------- |
| DraftKings  | `https://sportsbook.draftkings.com/leagues/basketball/nba` → click game → Player Props tab    |
| FanDuel     | `https://sportsbook.fanduel.com/navigation/nba` → click game → Player Props tab               |
| ESPN BET    | `https://espnbet.com/sport/basketball/organization/united-states/competition/nba`             |
| BetMGM      | `https://sports.betmgm.com/en/sports/basketball-7/betting/usa-9/nba-6004`                     |
| Caesars     | `https://sportsbook.caesars.com/us/bet/basketball/nba`                                        |
| Fanatics    | `https://sportsbook.fanatics.com/sports/basketball/nba`                                       |
| Hard Rock   | `https://app.hardrock.bet/home/sports/basketball/nba`                                         |

For non-NBA sports, substitute the league segment of the URL — e.g. `/sports/baseball/mlb`, `/sports/football/nfl`, `/sports/hockey/nhl`. The pattern is consistent within each book.

WebFetch prompt template for any Tier 1 page:

> "List all player props currently posted for {player_name} in tonight's {team} vs {opp} game. For each prop, return: prop_type, line, over price (American odds), under price (American odds). Skip props with no posted odds. If no props for this player are visible, say so."

**TIER 2 — REFERENCE SITES (for player_context._mean / _std, NOT for lines)**

- `basketball-reference.com/players/{letter}/{playerid}.html` — game logs, season averages
- `nba.com/stats/player/{nba_id}` — official splits
- `pro-football-reference.com`, `baseball-reference.com`, `hockey-reference.com` — same pattern other sports

Use these to compute `{prop}_mean` and `{prop}_std` from recent game logs (see §6.2.B / §6.2.C). Never inject a Tier 2 site as a market odds source.

**TIER 3 — AGGREGATORS (last resort, lines only)**

- Action Network odds pages, Covers consensus odds, SportsBookReview, VegasInsider

Acceptable when Tier 1 books are inaccessible or geo-restricted in the session. Always cross-verify against at least one Tier 1 book within the same turn if any Tier 1 page is reachable.

**BLOCK-LIST — NEVER use these as a source of `line`, `odds_over`, `odds_under`, or `player_context.*_mean`:**

- Dimers (`dimers.com`) — publishes its own projection model output, not current sportsbook lines. May be cited only for candidate discovery (which props are popular to bet).
- Lineups.com — same: projections + recommendations, not live market data.
- Rotowire / Yahoo / SI "pick" articles — narrative, not market.
- Any "best picks", "model edges", "expert plays" article — these are the AUTHOR's projections, which would corrupt the engine's calibration if injected as `_mean`.

Using a blocked source as `_mean` violates the no-estimated-values rule in §2. Using a blocked source as a line is a softer error but still wrong — the actual posted line may differ.

**STALE-LINE CHECK (mandatory before injecting any line)**

Before calling `inject_into_request(request, "odds_over", ...)`, verify the page's date stamp matches today's slate date.

- Sportsbook pages: look for the game time / tip-off displayed on the page. If it shows a past game time or yesterday's matchup, the page is stale (cached or a research-archive URL). Reject and try the next book.
- Reference pages: scan for "Last updated" or "Through {date}" footers.
- If a page has NO date stamp visible, do NOT trust it as a live line. Try one more book; if none have a visible date, exit per §6.3 condition 1 ("market not available").

Stale-line ingestion is the single highest-severity sourcing error because it produces engine output that LOOKS authoritative but reflects a market that no longer exists.

**CROSS-BOOK VERIFICATION (recommended for tier-A candidates)**

For any candidate you intend to ship as a Bet Card, fetch the same prop from a second Tier 1 book.

- Lines diverge by ≤ 0.5 and juice by ≤ 10 cents: solid market, proceed.
- Lines diverge by > 0.5 OR juice by > 15 cents: **soft market** — note in the Bet Card narrative; consider a lower confidence tier.
- One book has the prop, the other has removed it: late steam or scratch risk — investigate news before injecting.

------------------------------------------------------------
§6.2 — How to source each kind of missing slot
------------------------------------------------------------

For each `slot` in `missing_requirements`, identify what's missing and search accordingly:

A. **Market odds missing** (slots like `odds`, `odds_over`, `odds_under`, `odds.markets`):
   - Apply the §6.1.5 ranked recipe — Tier 1 books first, stale-line check mandatory, block-listed sources never as line input.
   - WebSearch query template (when you don't already know the right book URL): `"{player_name} {prop_type} {line} prop odds DraftKings FanDuel today"` or for games `"{home_team} vs {away_team} odds {league} today site:sportsbookreview.com OR site:vegasinsider.com"`.
   - Record the source URL AND the access timestamp AND the displayed game tip-off time for every line you inject.
   - If two books disagree by ≤ 5 cents (American odds), pick the median. If they disagree by more, flag the spread and pick the consensus (see cross-book verification in §6.1.5).
   - You must source `odds_over` AND `odds_under` from THE SAME book snapshot. Do not supply an "implied opposite" — the engine will reject it (see §B4 single-side handling). If only one side is posted, send only that side; the engine will return `edge_under: null, reason: "odds_unsourced"`.
   - If after 3 different searches across Tier 1 books you cannot find a market at all, exit the loop and declare "market not available — no public sportsbook is currently listing this prop." This is the ONLY acceptable exit without a Bet Card. Do NOT invent odds. Do NOT substitute a Dimers projection.

B. **Player mean missing** (slots like `player_context.{prop}_mean`):
   - WebSearch: `"{player_name} {prop_type} per game {current season} basketball-reference OR baseball-reference OR hockey-reference OR pro-football-reference"`.
   - Prefer official league sites (NBA.com, MLB.com, NHL.com, NFL.com, ESPN) and *-reference.com over generic aggregators.
   - If the season-long average and a recent rolling average disagree, prefer the most recent rolling window (last 5–10 games) for active props.
   - Inject as `player_context["{prop}_mean"] = float(value)`.

C. **Player std_dev missing** (slots like `player_context.{prop}_std`):
   - Preferred: WebSearch for the player's last 5–10 game logs, compute the sample standard deviation in Python.
   - Code template:

     ```python
     import statistics
     game_logs = [22, 17, 31, 25, 19, 28, 24, 33]   # values you scraped
     player_context[f"{prop_type}_std"] = statistics.stdev(game_logs)
     ```
   - Fallback only if game logs are unobtainable: apply a standard coefficient of variation: `std = 0.25 * mean`. This fallback MUST be disclosed to the user as "std estimated by coefficient-of-variation default; confidence may be capped at tier C by the quality gate."

D. **Team context missing** (slots like `home_context.off_rating`, `away_context.def_rating`):
   - WebSearch: `"{team} {league} offensive rating defensive rating {current season} site:basketball-reference.com"` (or sport equivalent).
   - For hockey/soccer/baseball, `off_rating` and `def_rating` are scaled offensive/defensive efficiency; if the league reports raw runs/goals-per-game or xG-for/xG-against, normalize per the archetype's `avg_total` if you have to, but PREFER the league's published rating fields.
   - Inject under the correct side: `home_context` or `away_context`.

E. **Tempo/pace missing** (slots like `home_context.pace`):
   - WebSearch the team's pace stat for the current season. Inject as `home_context["pace"]`.
   - If unobtainable, leave it empty — pace is OPTIONAL for most archetypes (only basketball lists it as required, where it is typically available from basketball-reference).

F. **Anything else (sport-specific)**:
   - For tennis: `serve_win_pct`, `return_win_pct` — search ATP/WTA player profiles.
   - For golf: `strokes_gained_total` — search Data Golf or the PGA stats page.
   - For fighting: `win_pct`, `finish_rate` — search the fighter's UFC.com or Tapology profile.
   - For esports: `map_win_rate` — search HLTV (CS), Liquipedia, or the league site.

§11 — CLV LIFECYCLE
================================================================================

CLV (closing-line value) is the difference between the price/line you took at decision time and the market's closing price/line. Beating the close is a leading indicator of long-term EV — measurable within hours of a game starting, far faster than realized ROI. CLV resolution requires TWO captures:

  1. **Decision-time capture** (§11.1) — the bet you actually took, recorded in the trace export block at analysis time.
  2. **JIT closing capture** (§11.2) — a snapshot of the market's closing price/line, sourced via WebFetch in no-local sessions or automated locally by Cowork with the Odds API adapter.

The local engine joins (1) and (2) on `(trace_id, market, selection_descriptor)` to compute CLV cents per bet. If either capture is missing, that bet is excluded from the CLV signal but its outcome still contributes to calibration.

------------------------------------------------------------
§11.1 — Decision-time capture (bet_record fields)
------------------------------------------------------------

You MUST capture three things on every Bet Card. If any are missing, the bet cannot be CLV-resolved:

1. The exact `line` and `odds` you took, attributed to a specific `book`.
2. The `decision_timestamp` so we can place "before the close" correctly.
3. A `selection_descriptor` that uniquely identifies the market (so the closing-line snapshot in §11.2 matches the right side).

Specifically:

- **Line and odds**: cite the sportsbook page you sourced from (per §8 source-citation rules). If the user is taking a different price than the one in your `Inputs used`, capture THEIR taken price in `bet_record.odds_taken`, not the one you used for engine input. The engine math runs on the price you fed it; CLV runs on the price the user actually executed.
- **Book**: must be one of the major US books (DraftKings, FanDuel, BetMGM, Caesars, ESPN BET, Hard Rock, PointsBet, Fanatics) or labeled "other:{book_name}" for offshore/sharp books.
- **Decision timestamp**: use the timestamp at which the user confirmed they were placing the bet, NOT the timestamp of your engine call. If the user is reviewing your Bet Card and will bet later, capture the timestamp when they confirm.
- **Selection descriptor**: a short canonical string the closing-line snapshot uses to match. Examples: `home_moneyline`, `away_spread_-3.5`, `total_over_226.5`, `Tatum_over_27.5_pts`. Use snake_case and include the line value where applicable. **Whatever descriptor you write in `bet_record` MUST be byte-identical to the descriptor you write in the §11.2 closing snapshot — the join is exact.**

If the user takes a bet at a price you cannot verify (e.g., a sharp book you cannot search), still record it — CLV resolution may fail for that single bet, but the calibration signal from the outcome remains valid.

You are NEVER allowed to FABRICATE a `decision_timestamp`, `book`, or `odds_taken`. If the user did not bet, `bet_record` is `null` — never invent a hypothetical bet to populate the block.

------------------------------------------------------------
§11.2 — JIT closing capture (just-in-time WebFetch protocol)
------------------------------------------------------------

In this no-local deployment, you cannot read the user's `.env` or call local repo automation. Closing lines are therefore sourced by YOU via WebFetch on the same sportsbook the user took the bet at, at approximately T-5 minutes before tip-off (or as close as the user prompts you to do so). In Cowork/local deployments, `OMEGA_COWORK.md` supersedes this section and should automate closing-line capture with the local Odds API adapter when available.

When the user asks you to capture closing lines (e.g., "grab the closing lines for tonight's pending bets"), you MUST:

1. Identify each pending bet that needs a closing snapshot. These are bets whose `decision_timestamp` is from the current session or a prior session, where you have NOT yet emitted a closing snapshot for the matching `(trace_id, market, selection_descriptor)`.
2. For each one, WebFetch the relevant sportsbook page (the book named in the `bet_record`) and read the current line/odds for the exact `selection_descriptor`.
3. Emit ONE JSON file per `trace_id` containing every line you captured for that trace. Multiple bets on the same game/trace_id share one file.

The block format:

```json
// SAVE AS: inbox/closing_lines/<trace_id>.json
{
  "trace_id": "sandbox-XXXX",
  "captured_at": "2026-05-15T23:55:00Z",
  "source": "draftkings.com",
  "lines": [
    {
      "market": "spread" | "total" | "moneyline" | "player_prop:pts" | ...,
      "selection_descriptor": "lakers_-3.5",
      "closing_line": -3.5,
      "closing_odds": -110
    }
  ]
}
```

Rules:

- `closing_line` is **required** for spread / total / player_prop markets. It is **optional** (use `null` or omit) for `moneyline`, which has no point.
- `closing_odds` is American odds (`-110`, `+145`, etc.). Always required, always numeric.
- `selection_descriptor` MUST be byte-identical to the descriptor used in the original `bet_record` (§11.1) — the join is exact.
- `source` should be a clean domain string (`draftkings.com`, `fanduel.com`, `betmgm.com`, etc.) — the same book whose price the user took, where possible.
- `captured_at` is the ISO 8601 UTC timestamp at which you ran the WebFetch.

If WebFetch fails (page changed, network error, market pulled) for a given selection, **OMIT THAT ENTRY**. Do not estimate. Do not fabricate. The user instruction line above the block:

> Save the JSON below to `inbox/closing_lines/<trace_id>.json` in your Omega repo and run `python scripts/ingest_closing_lines.py`.

If WebFetch fails for every line in the file, do not emit the block at all — explain in plain text which sources failed and offer to retry. The engine tolerates missing closes; what it does NOT tolerate is invented numbers.
